# Информативные коммиты

## Порядок работы

1. Проверить `git status` и `git diff --cached`; не включать чужие или несвязанные изменения.
2. При необходимости добавить только файлы текущей задачи. Не использовать `git add .`.
3. Сформулировать заголовок в формате `<type>(<scope>): <действие>` длиной до 72 символов.
4. Использовать `feat`, `fix`, `refactor`, `test`, `docs` или `chore`; scope указывать, когда он уточняет область.
5. Добавлять тело только для причины, риска, миграции или несовместимости — не более 2 коротких строк.
6. Перед коммитом повторно проверить staged diff, отсутствие секретов и совпадение сообщения с изменением.

## Правила сообщения

- Описывать результат изменения, а не процесс работы.
- Использовать глагол в начальной форме: `fix(auth): отклонить refresh JWT`.
- Не писать общие сообщения: `changes`, `fix`, `update`, `wip`.
- Не объединять независимые изменения в один commit.
- Не изменять историю (`--amend`, rebase, reset) без явного запроса.

## Примеры

- `fix(resume): фиксировать удаление резюме`
- `test(auth): покрыть отклонение refresh JWT`
- `docs: описать флаг test-аутентификации`
