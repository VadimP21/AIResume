# Поэтапная изоляция ORM-моделей через внутренние DTO

## Цель

Исключить SQLAlchemy-модели из взаимодействия:

```text
Router → Service → Repository
```

ORM-модели должны использоваться только внутри:

* `app/models/`;
* `app/db/models.py`;
* реализаций репозиториев;
* миграций и инфраструктурного кода.

Публичный HTTP API и схема PostgreSQL не должны измениться.

---

## Общие ограничения

* Не перезаписывать существующие незакоммиченные изменения.
* Не менять HTTP-контракты и response schemas.
* Не менять схему БД без крайней необходимости.
* Не выполнять массовый рефакторинг за один этап.
* После каждого этапа запускать проверки.
* Не переходить к следующему этапу при падающих тестах.
* Не исправлять несвязанные проблемы.
* Все выводы основывать только на существующем коде.

---

# Этап 0. Анализ и фиксация текущего состояния

Ничего не изменять.

1. Прочитать `AGENTS.md` и `CLAUDE.md`.
2. Найти все импорты `app.models` в:

   * `app/api`;
   * `app/services`;
   * dependencies;
   * тестах сервисов.
3. Найти все публичные методы сервисов и репозиториев, которые:

   * принимают ORM-модели;
   * возвращают ORM-модели;
   * используют ORM-типы в аннотациях.
4. Зафиксировать текущие HTTP-контракты.
5. Запустить базовые проверки:

```bash
ruff check .
ruff format --check .
mypy app
pytest
```

Подготовить отчёт:

* текущие нарушения границ;
* затрагиваемые файлы;
* риски;
* последовательность миграции.

Не писать код до завершения анализа.

# Этап 1. Создание базовой DTO-инфраструктуры

Создать пакет:

```text
app/dto/
├── __init__.py
├── common.py
├── users.py
├── resumes.py
└── versions.py
```

Использовать:

```python
@dataclass(frozen=True, slots=True)
```

## Требования к DTO

Разделить DTO по назначению:

### Команды

* `CreateUserCommand`
* `CreateResumeCommand`
* `UpdateResumeCommand`
* `CreateSectionCommand`
* `UpdateSectionCommand`

### DTO для чтения

* `UserDTO`
* `ResumeDTO`
* `ResumeSectionDTO`
* `ResumeVersionDTO`

### DTO для авторизации

Создать отдельный внутренний тип:

```text
UserAuthDTO
```

Он может содержать:

* `id`;
* `email`;
* `password_hash`;
* `is_active`;
* `token_version`.

Не включать `password_hash` в обычный `UserDTO`.

## Частичные обновления

Для update-команд различать:

* поле не передано;
* поле явно передано как `None`;
* поле передано со значением.

Использовать существующий Pydantic `model_fields_set`, собственный `UNSET` либо явный набор обновляемых полей.

Не подменять «не передано» значением `None`.

## SectionType

Переместить `SectionType` в независимый DTO или доменный модуль, например:

```text
app/dto/resumes.py
```

ORM-модель должна импортировать этот enum, а не наоборот.

## Проверка этапа

* DTO не импортируют SQLAlchemy.
* DTO не зависят от FastAPI и Pydantic HTTP-схем.
* Существующий код пока продолжает работать.

Запустить:

```bash
ruff check .
ruff format --check .
mypy app
pytest
```

# Этап 2. Добавление ORM → DTO мапперов

Создать явные функции преобразования внутри слоя репозиториев или отдельного инфраструктурного модуля:

```text
app/repositories/mappers/
```

Примеры:

* `user_to_dto()`;
* `user_to_auth_dto()`;
* `resume_to_dto()`;
* `section_to_dto()`;
* `version_to_dto()`.

## Правила

* ORM-объекты не должны покидать репозиторий.
* Все необходимые relationships должны быть загружены до преобразования.
* Не допускать lazy loading после выхода из репозитория.
* Использовать `selectinload()` или `joinedload()` по необходимости.
* Не создавать универсальный reflection-based mapper.
* Использовать явный и типизированный mapping.

## Особое внимание

Для `ResumeDTO` заранее загружать секции и другие данные, необходимые API или сервису.

Это должно исключить:

* `MissingGreenlet`;
* N+1;
* запросы во время Pydantic-сериализации.

## Проверка этапа

Добавить unit-тесты мапперов.

Запустить полный набор проверок.

# Этап 3. Вертикальный срез User/Auth

Сначала мигрировать только авторизацию и пользователей.

## UserRepository

Изменить публичные методы так, чтобы они:

* принимали UUID, строки, примитивы или команды;
* создавали и изменяли ORM-модели только внутри репозитория;
* возвращали `UserDTO` или `UserAuthDTO`;
* не возвращали SQLAlchemy-модели.

## AuthService

Изменить сервис так, чтобы он:

* не импортировал `app.models`;
* работал с `UserDTO`, `UserAuthDTO` и примитивами;
* возвращал DTO или токены;
* сохранял существующее поведение JWT, Redis и Argon2.

## Dependencies

`get_current_user` должен возвращать:

* предпочтительно UUID пользователя;
* либо безопасный `UserDTO`, если роуту действительно нужны данные пользователя.

Защищённым роутам не передавать ORM-модель.

## HTTP-слой

Pydantic request schemas преобразовывать в команды.

Pydantic response schemas строить из DTO.

## Проверка этапа

Проверить сценарии:

* регистрация;
* вход;
* `/refresh`;
* `/logout`;
* `/users/me`;
* неактивный пользователь;
* неверный токен;
* несовпадение `token_version`.

Запустить полный набор проверок.

# Этап 4. Вертикальный срез Resume CRUD

Мигрировать:

* `ResumeRepository`;
* `ResumeService`;
* resume dependencies;
* resume routers.

## ResumeRepository

Методы должны:

* принимать `user_id`, `resume_id` и команды;
* создавать и изменять ORM внутри репозитория;
* возвращать `ResumeDTO`;
* загружать секции до преобразования в DTO.

## ResumeService

Сервис должен:

* не импортировать ORM-модели;
* принимать команды и UUID;
* возвращать DTO;
* управлять транзакцией согласно текущим правилам проекта;
* сохранять текущие проверки ownership.

## Обновления

При PATCH/частичном обновлении не затирать поля, отсутствующие в запросе.

## Проверка этапа

Проверить:

* создание резюме;
* получение списка;
* получение одного резюме;
* обновление;
* удаление;
* ownership;
* отсутствие доступа к чужому резюме;
* неизменность HTTP-ответов.

Запустить полный набор проверок.

# Этап 5. Вертикальный срез Resume Sections

Мигрировать операции создания, редактирования, сортировки и удаления секций.

## Требования

* Использовать `SectionType` из независимого модуля.
* Репозиторий принимает команды и идентификаторы.
* Репозиторий возвращает `ResumeSectionDTO` или обновлённый `ResumeDTO`.
* JSONB-содержимое секции не должно утекать как ORM mutable state.
* Проверить копирование вложенных изменяемых структур при необходимости.

## Проверка этапа

Проверить:

* создание каждого типа секции;
* редактирование;
* удаление;
* изменение порядка;
* валидацию содержимого;
* ownership;
* сериализацию ответа.

# Этап 6. Вертикальный срез Versioning

Мигрировать:

* `ResumeVersionRepository`;
* `VersioningService`;
* snapshot logic.

## Требования

* Snapshot формировать из `ResumeDTO`, а не ORM.
* Не обращаться к lazy relationships.
* Snapshot должен сохранять текущий формат.
* `ResumeVersionDTO` должен быть неизменяемым.
* Не менять схему таблицы версий в рамках этого рефакторинга.

## Проверка этапа

Проверить:

* создание версии;
* получение списка версий;
* получение конкретной версии;
* восстановление версии, если оно реализовано;
* неизменность snapshot-структуры.

# Этап 7. Import, Export и Rendering

Перевести импорт, экспорт и генерацию документов на DTO.

## Требования

* HTML/PDF rendering получает `ResumeDTO`.
* Экспорт не принимает ORM-модели.
* Jinja2 и WeasyPrint работают с подготовленным DTO-контекстом.
* Импорт преобразует входные данные в команды.
* Инфраструктурный код не должен инициировать дополнительные ORM-запросы во время rendering.

## Проверка этапа

Проверить:

* HTML preview;
* PDF export;
* JSON import/export;
* отсутствие изменений публичного формата;
* отсутствие lazy loading.

# Этап 9. Очистка

После успешной миграции всех вертикальных срезов:

1. Удалить устаревшие методы репозиториев.
2. Удалить временные адаптеры.
3. Удалить неиспользуемые импорты ORM.
4. Проверить отсутствие дублирующих DTO.
5. Проверить, что DTO не являются механическими копиями всех ORM-моделей.
6. Оставить только DTO, реально пересекающие архитектурные границы.

Не выполнять дополнительный рефакторинг транзакций и исключений.
