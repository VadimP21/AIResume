# Миграции базы данных

## Цель

Безопасно изменить схему PostgreSQL через Alembic.

## Порядок работы

1. Изменить SQLAlchemy-модели в `app/models/`.
2. Убедиться, что изменение модели действительно требует миграции.
3. Запустить PostgreSQL и применить актуальную схему:

```powershell
docker compose up -d
uv run alembic upgrade head
```

4. Создать миграцию с осмысленным сообщением:

```powershell
uv run alembic revision --autogenerate -m "describe_change"
```

5. Вручную проверить созданный файл в `alembic/versions/`.
6. Применить миграцию:

```powershell
uv run alembic upgrade head
```

7. Запустить проверки из [сценария тестирования](testing.md).

## Что проверить в миграции

- `upgrade()` и `downgrade()` соответствуют друг другу;
- корректны типы, nullable-поля, значения по умолчанию и внешние ключи;
- индексы создаются для полей, используемых в фильтрации и связях;
- миграция не удаляет данные и не блокирует таблицы без явного согласования;
- нет изменений, не относящихся к задаче.

## Откат в локальной среде

После проверки `downgrade()` допустим локальный откат на одну ревизию:

```powershell
uv run alembic downgrade -1
uv run alembic upgrade head
```

Не выполнять откат в общей или production-среде без согласованного плана.

## Результат

В описании изменения указать ревизию, затронутые таблицы, наличие обратной совместимости и возможные риски для данных.
