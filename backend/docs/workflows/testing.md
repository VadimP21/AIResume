# Тестирование

## Цель

Проверить корректность изменения до ревью и объединения ветки.

## Проверки Python-кода

```powershell
uv run ruff check .
uv run ruff format --check .
uv run mypy app
uv run pytest
```

Для изменений только в Markdown проверить, что все относительные ссылки на
файлы существуют; запуск Python-проверок не требуется.

## Сценарии тестов

Для новой функциональности покрыть применимые случаи:

- успешный сценарий;
- ошибки валидации;
- аутентификация и авторизация;
- граничные случаи;
- ожидаемые ошибки внешних зависимостей.

Для endpoint-тестов дополнительно проверить HTTP-статус и схему ответа. Для изменений доступа к данным проверить, что запросы выполняются асинхронно и без N+1.

## При изменении схемы БД

До запуска тестов применить миграции:

```powershell
uv run alembic upgrade head
```

Если PostgreSQL или Redis нужны тесту, запустить инфраструктуру:

Перед запуском Docker создать `.env.docker` из `.env.docker.example` и заменить
все значения `CHANGE_ME_*`.

```powershell
docker compose up -d
```

Интеграционные тесты, проверяющие фиксацию транзакций в новой сессии, требуют
отдельную PostgreSQL-базу с применёнными миграциями и переменную
`TEST_DATABASE_URL` в формате `postgresql+asyncpg://...`. Не указывать рабочую
или локальную development-базу.

Redis integration-тесты требуют отдельный Redis и `TEST_REDIS_URL`. Не
использовать Redis из development или production.

## Результат

Зафиксировать точные команды и их статус. Падения тестов, не связанные с задачей, не исправлять без отдельного запроса.
