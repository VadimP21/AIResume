# Code review

## Цель

Проверить изменение до объединения ветки без внесения правок в рамках ревью.

## Порядок работы

1. Просмотреть diff и связанные изменения моделей, схем, миграций и тестов.
2. Проверить соответствие слоям `Router → Service → Repository`.
3. Выполнить проверки из [сценария тестирования](testing.md) или подтвердить их результат от автора.
4. Зафиксировать только подтверждённые по коду замечания.

## Чек-лист

### Корректность и архитектура

- Бизнес-логика находится в service, SQL — в repository.
- Router содержит только HTTP-логику, валидацию и dependency injection.
- Транзакции не управляются из repository.
- Публичные API и response-модели не изменены без задачи.

### Async и производительность

- Нет блокирующих вызовов в async-коде.
- Используется `AsyncSession` и SQLAlchemy 2.x API.
- Нет N+1, лишних запросов и загрузки всей таблицы без пагинации.
- Для связанных данных выбраны `selectinload()` или `joinedload()`, когда это требуется.

### Безопасность

- Проверены аутентификация и авторизация.
- Валидация входных данных выполняется схемами Pydantic.
- Не раскрываются секреты, пароли, JWT и внутренние ошибки.
- Нет конкатенации пользовательского ввода в SQL.

### Тесты и миграции

- Добавлены или обновлены тесты для нового поведения.
- Миграции проверены вручную, если изменена схема БД.
- `upgrade()` и `downgrade()` миграции согласованы.

## Формат замечания

Для каждого замечания указать:

- уровень: Critical, High, Medium или Low;
- файл и строку;
- описание проблемы и её влияние;
- конкретную рекомендацию.

Если подтверждённых замечаний нет, явно указать это и перечислить оставшиеся риски или непроверенные области.
