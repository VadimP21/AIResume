# Задача: одностраничный frontend для импорта и экспорта резюме

## Цель

Создать отдельное одностраничное frontend-приложение для AIResume Backend.

Основные функции:

* авторизация пользователя;
* выбор существующего резюме;
* импорт резюме из JSON-файла;
* экспорт резюме в JSON;
* экспорт резюме в PDF, если endpoint поддерживается backend;
* отображение результата и ошибок операций.

## Технологии

Использовать:

* React;
* TypeScript;
* Vite;
* стандартный `fetch` или существующий HTTP-клиент;
* CSS Modules либо обычный CSS;
* Vitest;
* React Testing Library.

Не добавлять тяжёлые UI-библиотеки без необходимости.

## Расположение

Создать frontend как отдельное приложение:

```text
AIResume/
├── backend/
└── frontend/
```

Не изменять backend без крайней необходимости.

## Этап 1. Анализ

Сначала ничего не изменять.

1. Прочитать документацию проекта.
2. Изучить OpenAPI FastAPI.
3. Найти реальные endpoints:

   * авторизация;
   * refresh token;
   * список резюме;
   * получение резюме;
   * импорт;
   * экспорт JSON;
   * экспорт PDF.
4. Определить форматы запросов и ответов.
5. Проверить настройки CORS.
6. Подготовить краткий план реализации.
7. Перечислить создаваемые и изменяемые файлы.

Не начинать реализацию до завершения анализа.

## Этап 2. Архитектура frontend

Использовать структуру:

```text
frontend/src/
├── api/
│   ├── client.ts
│   ├── auth.ts
│   └── resumes.ts
├── components/
│   ├── AuthForm.tsx
│   ├── ResumeSelector.tsx
│   ├── ImportPanel.tsx
│   ├── ExportPanel.tsx
│   ├── StatusMessage.tsx
│   └── LoadingIndicator.tsx
├── hooks/
├── types/
├── App.tsx
├── main.tsx
└── styles/
```

Компоненты должны быть небольшими и отвечать только за одну задачу.

## Этап 3. Интерфейс

Создать одну страницу со следующими блоками:

### Авторизация

* email;
* пароль;
* кнопка входа;
* отображение ошибки;
* выход из аккаунта.

### Выбор резюме

* загрузка списка резюме;
* выбор резюме;
* отображение названия и идентификатора.

### Импорт

* выбор `.json` файла;
* проверка расширения и JSON-синтаксиса;
* отправка данных в backend;
* отображение успешного результата или ошибки.

### Экспорт

Кнопки:

* «Экспортировать JSON»;
* «Экспортировать PDF», если endpoint существует.

Файл должен автоматически скачиваться через браузер.

## Этап 4. API-клиент

Требования:

* базовый URL брать из `VITE_API_URL`;
* не хардкодить адрес backend;
* добавлять access token в `Authorization: Bearer`;
* корректно обрабатывать JSON и файловые ответы;
* обрабатывать `401`, `403`, `404`, `422` и `500`;
* не выводить токены и персональные данные в консоль.

Создать:

```text
frontend/.env.example
```

Пример:

```env
VITE_API_URL=http://localhost:8000/api/v1
```

Не создавать `.env` с реальными секретами.

## Этап 5. Работа с токенами

Следовать существующему backend-контракту.

Для MVP:

* access token хранить только настолько долго, насколько необходимо приложению;
* не помещать секреты в исходный код;
* при наличии рабочего refresh endpoint реализовать обновление токена;
* при неуспешном refresh завершать сессию пользователя.

Не менять backend-механику авторизации самостоятельно.

## Этап 6. UX

Интерфейс должен быть:

* на русском языке;
* адаптивным;
* понятным без инструкции;
* доступным с клавиатуры;
* с disabled-состояниями кнопок;
* с индикатором загрузки;
* с понятными сообщениями об ошибках.

Не использовать декоративные элементы, не связанные с задачей.

## Этап 7. Тесты

Добавить тесты для основных сценариев:

* успешный вход;
* ошибка авторизации;
* загрузка списка резюме;
* выбор резюме;
* проверка некорректного JSON;
* успешный импорт;
* ошибка импорта;
* скачивание JSON;
* скачивание PDF;
* обработка сетевой ошибки.

API-запросы в тестах мокировать.

## Этап 8. Проверка

Выполнить:

```bash
npm install
npm run lint
npm run test
npm run build
```

Убедиться, что production build создаётся без ошибок.

## Ограничения

* Не менять публичные API backend.
* Не изменять схему PostgreSQL.
* Не добавлять регистрацию, редактор резюме и AI-функции.
* Не добавлять роутинг, если приложение состоит из одной страницы.
* Не использовать Redux без реальной необходимости.
* Не перезаписывать незакоммиченные изменения.
* Не придумывать endpoints — использовать только обнаруженные в backend.

## Критерии готовности

Функциональность считается готовой, если пользователь может:

1. Авторизоваться.
2. Загрузить список своих резюме.
3. Выбрать резюме.
4. Импортировать корректный JSON.
5. Скачать резюме в JSON.
6. Скачать PDF, если backend это поддерживает.
7. Получить понятное сообщение при любой ошибке.

## Формат итогового отчёта

После реализации показать:

* созданные и изменённые файлы;
* использованные endpoints;
* команды запуска;
* результаты lint, tests и build;
* известные ограничения;
* необходимые ручные проверки.
