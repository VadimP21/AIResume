# Этап 1: базовая DTO-инфраструктура

## Цель

Создать внутренние DTO для постепенной изоляции SQLAlchemy ORM-моделей. На этом
этапе существующие API, ORM-модели, репозитории и сервисы не изменяются.

## Состав пакета

Создаётся пакет `app/dto/`:

* `common.py` — singleton `UNSET` и его тип для различения отсутствующего поля,
  `None` и переданного значения в командах частичного обновления.
* `users.py` — `CreateUserCommand`, `UserDTO`, `UserAuthDTO`.
* `resumes.py` — независимый `SectionType`, команды создания и обновления
  резюме/секций, `ResumeDTO`, `ResumeSectionDTO`.
* `versions.py` — `ResumeVersionDTO`.
* `__init__.py` — реэкспорт публичного DTO API.

Все DTO оформляются как `@dataclass(frozen=True, slots=True)` и не импортируют
SQLAlchemy, FastAPI или Pydantic HTTP-схемы.

## Частичные обновления

`UpdateResumeCommand` и `UpdateSectionCommand` используют `UNSET` как значение
по умолчанию. Это сохраняет три состояния поля:

* `UNSET` — поле не передано;
* `None` — поле явно передано как `null`;
* значение — поле передано с конкретным значением.

Конвертация Pydantic-схем на этом этапе не добавляется; она будет выполнена при
миграции соответствующих вертикальных срезов.

## Данные и границы

`UserDTO` не содержит хеш пароля. `UserAuthDTO` содержит только данные,
необходимые для аутентификации: идентификатор, email, хеш пароля, активность и
версию токена. `SectionType` выносится в DTO-модуль, но ORM-модель пока не
изменяется, поэтому временно сохраняется существующее определение enum.

## Проверки

Перед реализацией добавляются unit-тесты, которые должны завершиться ошибкой из-за
отсутствия пакета `app.dto`. После реализации они проверяют неизменяемость DTO,
контракт полей и различение состояний частичного обновления. Затем запускаются
Ruff, mypy и полный pytest.

## Ограничения

HTTP-контракты, response schemas, схема PostgreSQL и текущая бизнес-логика не
меняются. Незакоммиченные изменения пользователя не перезаписываются.
